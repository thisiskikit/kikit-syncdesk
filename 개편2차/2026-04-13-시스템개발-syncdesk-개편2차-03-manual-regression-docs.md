# 개편 2차-3 — 운영 회귀 검증 문서화

## 작업 목적
브라우저 수동 검증 시나리오를 운영 자산으로 고정하고, 출고와 작업센터의 회귀 위험을 관리할 수 있게 문서화합니다.

## 반드시 먼저 읽을 문서
- `2026-04-13-시스템개발-syncdesk-개편2차-00-common-rules.md`
- `docs/current-status.md`

## 작업 목표
현재 `codex/dev`의 실제 동작 기준으로 수동 검증 시나리오와 회귀 체크리스트를 정리합니다.

## 반드시 달성할 것
1. 아래 흐름을 manual regression 문서로 정리합니다.
   - dashboard -> fulfillment 진입
   - 빠른 수집 후 신규 주문 집중 보기
   - 혼합 선택 후 자동 제외 실행
   - fulfillment drawer 확인 흐름
   - work-center 복구 흐름
2. 각 단계마다 기대 결과를 씁니다.
3. 회귀 위험도를 `high / medium / low`로 나눕니다.
4. 향후 자동화 테스트 후보를 별도 섹션으로 정리합니다.
5. 아래 문서를 함께 갱신합니다.
   - `docs/current-status.md`
   - `docs/change-log.md`

## 권장 생성 파일
- `docs/qa/manual-fulfillment-regression.md`
- `docs/qa/work-center-recovery-scenarios.md`

## 문서 작성 원칙
- 사람이 바로 따라 할 수 있게 짧고 단계형으로 씁니다.
- 화려한 설명보다 확인 행동과 기대 결과가 먼저입니다.
- 확인하지 못한 것은 `추정`으로 표시합니다.

## 하지 말 것
- 근거 없는 자동화 테스트 추가
- 현재 동작과 맞지 않는 이상적 시나리오 작성
- 미검증 내용을 사실처럼 단정

## 완료 기준
- 운영자가 실제로 따라 할 수 있는 수동 시나리오가 생깁니다.
- high-risk 회귀 포인트가 분리됩니다.
- 자동화 테스트 후보가 정리됩니다.
- docs에 신규 QA 문서 반영 사실이 남습니다.
