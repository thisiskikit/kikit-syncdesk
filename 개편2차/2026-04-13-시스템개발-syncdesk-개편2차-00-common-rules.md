# 공통 운영 규칙 — SyncDesk 개편 2차

## 목적
이 문서는 `kikit-syncdesk`의 `codex/dev` 브랜치에서 개편 2차 작업을 수행할 때 모든 작업 프롬프트 앞에 공통으로 적용할 운영 규칙입니다.

## 공통 규칙
- 한국어 UI를 유지합니다.
- 현재 기준 브랜치는 `codex/dev`입니다.
- 이번 차수의 목표는 **새 기능 확장**이 아니라 **운영 구조 고정**입니다.
- 코드 변경과 문서 변경은 반드시 같은 작업 세트로 처리합니다.
- 아래 문서는 작업 시 항상 함께 갱신합니다.
  - `docs/current-status.md`
  - `docs/change-log.md`
  - `docs/structure-overview.md`
- 검증하지 못한 내용은 완료로 단정하지 말고 반드시 `추정`으로 표시합니다.
- 기존 동작을 함부로 깨지 않습니다.
- 기존 route, API shape, data shape는 꼭 필요한 경우가 아니면 유지합니다.
- 레거시를 치우더라도 즉시 삭제보다 **메인 노출 제거 + 얇은 이행 구조 유지**를 우선합니다.
- 디자인 polish보다 먼저, 실제 운영 행동 흐름이 자연스러운지 확인합니다.
- 구현 완료 전 반드시 아래를 남깁니다.
  - 변경 파일 목록
  - 구조 변경 요약
  - 테스트 결과
  - 직접 검증 못 한 항목
  - 남은 위험
  - docs 반영 내용

## 2차 범위
이번 개편 2차의 허용 범위는 아래 3축입니다.
1. 출고 화면 coordinator 분해
2. 작업센터 recovery-first 강화
3. 운영 회귀 검증 문서화

## 2차 금지사항
- CS 통합 실행 화면 신설
- 채널 세부 화면 전면 개편
- 상품/가격 기능 재확장
- 디자인 토큰 대공사
- 모바일 완성형 대응
- unrelated page 리디자인

## 작업 원칙
### 원칙 1. 출고가 최우선
현재 제품의 핵심은 출고입니다.  
출고 화면이 큰 coordinator 상태로 남아 있으면 이후 모든 수정 비용이 커집니다.

### 원칙 2. 작업센터는 로그 뷰어가 아니라 복구 콘솔
작업센터는 “무슨 로그가 있었는가”보다 “지금 무엇을 복구해야 하는가”가 먼저 보여야 합니다.

### 원칙 3. 문서도 운영 자산
QA 문서와 current-status 문서는 부수 산출물이 아니라 운영 자산입니다.

## 응답 형식
각 작업 완료 보고는 아래 순서로 작성합니다.
1. 변경한 파일 목록
2. 무엇을 왜 바꿨는지
3. 유지된 동작 / 바뀐 내부 구조
4. 테스트 및 검증 결과
5. 직접 검증하지 못한 항목
6. 남은 리스크
7. docs 반영 내용
