# Work Center Recovery Scenarios

- 기준 날짜: 2026-04-13
- 기준 브랜치: `codex/dev`
- 목적: 작업센터의 recovery-first 흐름을 수동 회귀 시나리오로 고정합니다.
- 주의: 브라우저에서 아직 직접 검증하지 못한 기대 결과는 `추정`으로 표시합니다.

## 사전 조건

- `/work-center`에 최근 작업 로그가 최소 몇 건 존재해야 합니다.
- 가능하면 `재시도 가능 실패`, `수동 확인 실패`, `경고 또는 느린 요청`이 각각 1건 이상 있는 상태가 좋습니다.

## 시나리오 1. 복구 묶음과 리스트 우선순위 확인

- 위험도: high
- 단계:
1. `/work-center?tab=operations`에 진입합니다.
2. 상단 metric과 `복구 묶음` 카드를 확인합니다.
3. 첫 5개 row가 어떤 순서로 정렬되는지 확인합니다.
- 기대 결과:
  - 상단 metric은 `즉시 재시도 / 원인 확인 / 진행 관찰 / 완료 / 느린 요청` 기준으로 읽힙니다.
  - `복구 묶음`에는 같은 성격 실패가 묶여 count와 영향 범위가 함께 보입니다.
  - 리스트 상단에는 재시도 가능한 실패가 먼저 오고, 그 다음 수동 확인이 필요한 실패/경고가 옵니다.
  - `추정`: 실제 운영 데이터에서도 동일한 오류 요약은 같은 묶음으로 합쳐질 가능성이 높습니다.

## 시나리오 2. 리스트에서 상세 패널과 재시도 동선 확인

- 위험도: high
- 단계:
1. 재시도 가능한 operation row를 하나 선택합니다.
2. row 우측의 `상세` 또는 row 클릭으로 detail overlay를 엽니다.
3. `지금 할 일` 카드와 `영향 범위` 카드를 확인합니다.
4. 상세 안의 `재시도` 버튼을 눌러 요청을 보냅니다.
- 기대 결과:
  - `지금 할 일` 카드에 `즉시 재시도` 또는 그에 준하는 recovery label이 보입니다.
  - `영향 범위` 카드에 대상 건수, 기록된 티켓 수, 실패/경고 개수가 보입니다.
  - 재시도 가능 작업이면 상세 안에서도 즉시 재시도 버튼이 노출됩니다.
  - `추정`: 재시도 성공 시 작업 상태 패널에 로컬 operation toast가 남고, 목록이 새로고침됩니다.

## 시나리오 3. 상세 foldout 확인

- 위험도: medium
- 단계:
1. 작업 상세 overlay에서 `작업 티켓 상세`를 펼칩니다.
2. `요청 / 결과 요약`, `원본 요청 payload`, `정규화 payload`를 차례로 확인합니다.
3. overlay를 닫고 목록으로 복귀합니다.
- 기대 결과:
  - 티켓 상세에는 최대 5건까지 실패/경고/건너뜀 우선 샘플이 보입니다.
  - payload 관련 정보는 foldout 안에만 있고, 메인 목록에는 과하게 노출되지 않습니다.
  - 닫기 후에는 현재 필터와 선택 문맥을 유지한 채 목록으로 돌아옵니다.

## 시나리오 4. 이벤트 탭 관찰 흐름

- 위험도: low
- 단계:
1. 상단 탭을 `시스템 / 성능`으로 바꿉니다.
2. 경고 또는 느린 요청 로그를 하나 선택합니다.
3. 이벤트 메타 섹션을 확인합니다.
- 기대 결과:
  - 이벤트 탭은 재시도보다 관찰 중심 문구로 읽혀야 합니다.
  - event row는 `시스템 확인`, `경고 관찰`, `성능 관찰` 같은 lane 문구로 요약됩니다.
  - 이벤트 상세에서는 payload 대신 `이벤트 메타`가 기본 확인 대상입니다.

## High-Risk 회귀 포인트

- high: retryable 실패가 리스트 상단에서 밀려나지 않는지
- high: 상세 패널 `재시도` 버튼이 retryable 여부와 정확히 연동되는지
- high: `복구 묶음` count와 실제 리스트 항목 수가 크게 어긋나지 않는지
- medium: 티켓 샘플이 실패/경고/건너뜀 우선 규칙을 유지하는지
- low: 이벤트 탭의 관찰 lane 문구가 operation 탭과 혼동되지 않는지

## 자동화 테스트 후보

- recovery helper의 lane 계산과 grouping 규칙을 unit test로 확대
- work-center list 정렬 우선순위를 React integration test로 고정
- detail overlay에서 retry button 노출/비노출 조건을 component test로 검증
- browser automation으로 `복구 묶음 -> detail -> retry -> refresh` 흐름을 smoke test화
