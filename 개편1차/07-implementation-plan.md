# 07. Implementation Plan

## 문서 목적
이 문서는 `KIKIT SyncDesk` UI 개편 1차의 구현 순서, 범위, 리스크, QA 포인트를 정리합니다.  
목표는 **한 번에 모든 것을 바꾸지 않고, 운영 데스크 정체성을 가장 강하게 느끼게 하는 최소 핵심 축부터 안전하게 적용**하는 것입니다.

---

## 1) 이번 차수 범위
### 포함 범위
- App Shell / Navigation 재정렬
- Dashboard 리디자인
- Fulfillment 리디자인
- 공통 출고 판단 상태/사유 체계 반영
- 공통 Badge / Label / Drawer 패턴 반영
- 레거시 노출 제거
- 작업센터 명칭/역할 정리
- 관련 docs 업데이트

### 제외 범위
- CS 전체 화면 풀 리디자인
- 모바일 완성형 리디자인
- 채널 세부 화면 전면 재설계
- 정산/통계 고도화
- 상품/가격 기능 고도화
- 전체 design token system 리팩터링

---

## 2) 구현 우선순위
이번 차수는 아래 순서로 진행합니다.

### Phase 1. 해석 고정 + Navigation 정리
목표:
- 운영 데스크 정체성을 UI 시작점에서 느끼게 만들기

작업:
- 상위 메뉴를 `대시보드 / 출고 / CS / 채널 / 작업센터 / 설정`으로 재배치
- bulk-price / product-edit / grouped products 메인 노출 제거
- 구 라우트에서 새 구조로 redirect 또는 wrapper 설계
- 설명 문구 및 empty state 방향성 정리

---

### Phase 2. Dashboard 리디자인
목표:
- 앱 첫 화면을 “오늘 작업 시작점”으로 재설계

작업:
- 헤더 영역
- 5개 핵심 액션 카드
- 우선순위 패널
- 최근 이슈 피드
- 빠른 작업 패널
- 하단 상태 스트립

---

### Phase 3. Fulfillment 리디자인
목표:
- 출고 화면을 “판정 + 실행 + 예외 확인” 중심 화면으로 재구성

작업:
- 상태 탭 재정의
- 필터 바 재구성
- 결과 요약 바
- 메인 테이블 컬럼 정리
- 우측 Drawer 패턴 적용
- 일괄 작업 바 조건부 노출
- 송장/재확인 액션 정리

---

### Phase 4. Shared UI / State Model
목표:
- 출고 판단 상태와 사유 체계를 공통 컴포넌트로 통합

작업:
- DecisionBadge
- ReasonLabel
- SummaryBar
- DrawerSection
- Reason-aware action menu
- 상태별 버튼 허용 규칙 적용

---

### Phase 5. Work Center / Channel / Docs Alignment
목표:
- 1차 핵심 화면 이후 남은 정체성 충돌 제거

작업:
- 작업센터 명칭과 설명 문구 정리
- 채널 화면이 메인 동선을 방해하지 않도록 재배치
- docs 일괄 업데이트
- 미적용 범위와 후속 과제 명시

---

## 3) 기술 구현 원칙
### 원칙 A. 얇은 라우트 이행
기존 라우트를 즉시 삭제하지 말고,  
새 구조로 유도하는 redirect / wrapper를 먼저 둡니다.

### 원칙 B. Shared 컴포넌트 우선
Dashboard와 Fulfillment가 공통으로 쓰는 Badge/Label/Empty State/Drawer 패턴을 먼저 정리합니다.

### 원칙 C. 행동 흐름 우선
디자인 polish보다 먼저,
- 첫 클릭
- 필터링
- 상세 확인
- 액션 실행
의 흐름이 자연스러운지 확인합니다.

### 원칙 D. 문서 동시 갱신
코드 변경과 문서 변경은 반드시 같은 작업 세트로 처리합니다.

---

## 4) 레거시 처리 방침
### 즉시 메인 노출 제거
- bulk-price
- product-edit
- grouped products

### 완전 삭제 여부
이번 차수에서는 코드 완전 삭제보다 **메인 노출 제거 + 구 라우트 정리**를 우선합니다.

### 이유
- 리스크를 낮춤
- 기능 의존성 확인 시간을 확보
- 운영 데스크 중심 UX를 먼저 확정

---

## 5) 주요 리스크
### 리스크 1. 겉모습만 바뀌고 구조는 안 바뀌는 경우
회피:
- IA 변경과 라우트 재귀속을 포함해서 작업
- 메뉴명만 바꾸고 내부가 그대로인 상태 금지

### 리스크 2. 테이블에 정보가 다시 과하게 붙는 경우
회피:
- 목록과 Drawer 역할 분리 문서 기준 유지
- 원본/로그/CS 상세는 Drawer로 후퇴

### 리스크 3. 상태/사유 체계가 화면마다 다르게 쓰이는 경우
회피:
- `05-decision-model.md`를 단일 기준 문서로 사용
- 공통 컴포넌트화

### 리스크 4. 문서 업데이트 누락
회피:
- 변경사항 커밋 전 docs 체크리스트 필수 적용

## 6) QA 체크포인트
### IA / Navigation
- 사용자가 앱을 열었을 때 “운영 데스크”로 느끼는가?
- 쿠팡/네이버가 상위 동선이 아니라 보조 레이어로 보이는가?

### Dashboard
- 핵심 카드가 행동으로 이어지는가?
- 우선순위 패널이 실질적으로 유용한가?
- 최근 이슈가 작업센터를 대체하지 않고 보조하는가?

### Fulfillment
- 첫 3초 안에 출고 판단이 보이는가?
- 송장 대기와 차단이 명확히 구분되는가?
- Drawer가 판단 확인 흐름을 방해하지 않는가?
- 일괄 액션이 위험 상태에서 잘 제한되는가?

### Shared UI
- 상태 색이 일관되는가?
- 사유 라벨이 지나치게 길지 않은가?
- 빈 상태 문구가 업무 문장인가?

### Docs
- current-status / change-log / structure-overview가 변경 사실을 반영하는가?
- 미적용 범위가 정직하게 남아 있는가?

---

## 7) 완료 기준
이번 차수는 아래가 만족되면 완료로 봅니다.

- Navigation이 운영 데스크 기준으로 재정렬됨
- Dashboard가 운영 시작점으로 동작함
- Fulfillment가 판정 + 실행 화면으로 재설계됨
- 상태/사유 체계가 공통 언어로 적용됨
- 레거시 노출이 메인 동선에서 제거됨
- 관련 docs가 함께 갱신됨
