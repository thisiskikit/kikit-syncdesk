# 04. Fulfillment Redesign

## 문서 목적
이 문서는 `출고` 화면을 **단순 주문 목록 화면**이 아니라,  
**출고 판단 + 송장 실행 + 예외 확인**을 위한 핵심 작업 화면으로 재설계하기 위한 기준을 정의합니다.

---

## 1) 출고 화면의 역할
### 잘못된 해석
- 신규 주문을 나열하는 목록 화면
- 송장 입력 화면
- 채널별 주문 데이터 표

### 올바른 해석
**출고 화면은 수집된 주문을 모아 출고 가능 여부를 판단하고, 송장을 반영하며, 차단/보류/재확인 필요 건을 안전하게 처리하는 운영 작업 공간입니다.**

---

## 2) 핵심 질문
출고 화면은 아래 질문에 가장 빠르게 답해야 합니다.

> 이 주문을 지금 보내도 되는가?

이 질문보다 먼저 원본 상태, 긴 이력, 개발자용 필드는 나오면 안 됩니다.

---

## 3) 전체 구조
출고 화면은 아래 7개 블록으로 구성합니다.

1. 헤더
2. 상태 탭
3. 필터 바
4. 결과 요약 바
5. 메인 테이블
6. 일괄 작업 바
7. 우측 Drawer

---

## 4) 블록별 상세 설계

## A. 헤더
### 구성
- 제목: `출고 관리`
- 설명: `수집된 주문을 확인하고, 출고 가능 여부를 판단한 뒤 송장을 반영합니다.`
- 우측 버튼:
  - `빠른 수집`
  - `송장 입력`

### 의도
이 화면이 단순 조회가 아니라 **판단 + 실행** 화면이라는 점을 박아줍니다.

---

## B. 상태 탭
### 최종 탭 구조
- 전체
- 오늘 출고
- 송장 대기
- 차단됨
- 확인 필요

### 고급 필터 내부 세분화
- 출고 가능
- 송장 대기
- 보류
- 차단
- 재확인 필요

### 의도
상단 탭은 실무자가 자주 보는 상위 업무 상태만 간결하게 보여주고,  
세부 상태는 필터로 확장합니다.

---

## C. 필터 바
### 기본 필터
- 채널
- 날짜
- 출고 판단
- 검색

### 고급 필터
- 사유
- CS 영향 여부
- 주문 상태
- 송장 상태
- 담당자/메모
- 최근 작업 결과

### 원칙
- 자주 쓰는 필터만 기본 노출
- 나머지는 `고급 필터`로 접기
- 필터 변경 시 결과 요약 바 즉시 반영

---

## D. 결과 요약 바
### 예시
- 총 184건
- 출고 가능 142건
- 송장 대기 17건
- 차단 21건
- 재확인 필요 4건

### 의도
현재 보고 있는 모수와 상태 분포를 바로 이해하게 합니다.

---

## E. 메인 테이블
### 기본 컬럼
1. 선택 체크박스
2. 상태
3. 주문번호
4. 상품명
5. 수량
6. 고객명
7. 채널
8. 주문시각
9. 출고 판단
10. 액션

### 컬럼 설계 원칙

#### 상태
주문 또는 진행 대표 상태를 짧게 표현  
예: `주문 접수`, `송장 반영중`, `발송 완료`

#### 주문번호
- monospace 또는 우측 정렬 친화 스타일
- 복사 액션 제공 가능

#### 상품명
- 1~2줄 제한
- 길면 말줄임 처리
- 옵션은 보조 텍스트 또는 Drawer로 이동

#### 수량
- 우측 정렬
- 스캔 친화적 표시

#### 고객명
- 최소 식별 정보만
- 전체 주소는 목록에서 제거

#### 채널
- 작은 badge
- 큰 면적 채널 컬러 금지

#### 주문시각
- 상대 시각 + hover 시 정밀 시각

#### 출고 판단
- 핵심 컬럼
- 값: 출고 가능 / 송장 대기 / 보류 / 차단 / 재확인 필요
- 바로 아래 보조 사유 1개 표시 가능

#### 액션
- 상세
- 송장 입력
- 더보기

### 목록에 두지 말 것
- 원본 JSON
- 상세 CS 타임라인
- 내부 에러 전문
- 긴 메모
- 전체 주소
- 플랫폼 응답 필드
- 디버그 상태 문자열

---

## F. 일괄 작업 바
### 노출 조건
체크박스로 1개 이상 선택 시에만 나타남

### 액션 후보
- 송장 입력
- 보류 처리
- 재확인
- 재수집

### 원칙
- 항상 노출하지 않음
- 선택된 항목들의 상태에 따라 허용 액션 제한
- 차단 상태에는 출고/송장 액션 비활성화

---

## G. 우측 Drawer
### 역할
목록은 얇게, 상세는 깊게 보여주는 핵심 상세 공간

### 상단
- 주문번호
- 채널 배지
- 출고 판단 상태 배지
- 빠른 액션:
  - 송장 입력
  - 보류 처리
  - 작업 이력 보기

### 섹션 구조
1. 주문 정보
2. 상품 정보
3. 출고 판단
4. CS 영향
5. 송장 상태
6. 작업 로그
7. 원본 데이터

### 섹션별 상세

#### 1. 주문 정보
- 주문번호
- 주문시각
- 고객명
- 수령지 요약
- 채널

#### 2. 상품 정보
- 상품명
- 옵션
- 수량
- 묶음 여부

#### 3. 출고 판단
- 현재 판단 상태
- 사유
- 설명 문구
- 운영 메모
- 최근 판정 시각

#### 4. CS 영향
- 취소 요청 여부
- 반품/교환 여부
- 문의 영향 여부
- 출고중지 여부

#### 5. 송장 상태
- 입력 여부
- 반영 상태
- 최근 반영 시도 결과

#### 6. 작업 로그
- 최근 수집
- 최근 동기화
- 최근 실패
- 재시도 이력

#### 7. 원본 데이터
- 아코디언
- JSON 보기
- 채널 세부 보기 버튼

### 섹션 순서 원칙
운영자가 실제로 보는 순서와 같아야 합니다.
`주문 → 상품 → 판단 → CS → 송장 → 로그 → 원본`

## 5) 행 단위 정보 계층
### 1순위
- 출고 판단
- 사유
- 차단/보류 여부

### 2순위
- 주문번호
- 상품명
- 수량
- 고객명

### 3순위
- 채널
- 주문시각

### 4순위
- 원본 상태
- 긴 이력
- 디버그 정보

---

## 6) 주요 액션 흐름
### 흐름 1. 오늘 출고 처리
대시보드 → 오늘 출고  
→ 출고 화면 `오늘 출고`  
→ row 선택  
→ Drawer 확인  
→ 송장 입력

### 흐름 2. 차단 건 확인
대시보드 → 출고 차단  
→ 출고 화면 `차단됨`  
→ row 클릭  
→ Drawer에서 차단 사유 확인  
→ 필요 시 CS로 이동

### 흐름 3. 재확인 필요 해결
대시보드 또는 출고 화면  
→ `재확인 필요` 또는 `확인 필요` 필터  
→ row 클릭  
→ Drawer 작업 로그 / 원본 상태 확인  
→ 재수집 또는 재확인

---

## 7) 출고 화면 금지사항
- 채널 탭을 상단 핵심 탭으로 두지 말 것
- 원본 상태를 핵심 판단보다 먼저 보여주지 말 것
- 행 액션을 과하게 노출하지 말 것
- 모든 상세 정보를 테이블 컬럼으로 올리지 말 것
- 레거시 상품/가격 도구 버튼을 이 화면에 끼워 넣지 말 것

---

## 8) 성공 기준
- 출고 화면에서 첫 3초 안에 `출고 가능`, `차단`, `확인 필요`가 보인다.
- 송장 작업이 자연스럽게 이어진다.
- 위험 주문은 정상 주문보다 더 또렷하게 드러난다.
- 목록과 상세가 명확히 분리된다.
- 사용자가 출고 화면을 “판정 화면”으로 인식한다.
