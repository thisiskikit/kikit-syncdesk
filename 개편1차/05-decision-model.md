# 05. Decision Model

## 문서 목적
이 문서는 `KIKIT SyncDesk` UI 개편의 핵심 운영 언어인  
**출고 판단 상태 체계**와 **사유 체계**를 정의합니다.

이 문서는 Dashboard, Fulfillment, Drawer, Badge, Filter, Summary Bar, Bulk Action의 기준 문서입니다.

---

## 1) 왜 별도 상태 체계가 필요한가
운영자는 원본 주문 상태, 송장 상태, CS 상태, 내부 작업 상태를 각각 보고 싶지 않습니다.  
운영자는 결국 아래 질문에 답만 빠르게 얻고 싶습니다.

> 그래서 이 주문을 지금 보내도 되는가?

따라서 원본 상태를 그대로 노출하지 않고,  
운영 관점으로 재해석한 **출고 판단 상태**를 UI 중심 언어로 사용합니다.

---

## 2) 최종 출고 판단 상태
최종 상태는 5개로 고정합니다.

1. 출고 가능
2. 송장 대기
3. 보류
4. 차단
5. 재확인 필요

---

## 3) 상태별 정의

## 1. 출고 가능
### 의미
지금 기준으로 바로 송장 입력/출고 진행이 가능한 상태

### 예시
- 주문 정상
- CS 영향 없음
- 취소/반품/교환/출고중지 없음
- 강한 시스템 오류 없음

### 라벨
`출고 가능`

### 설명 문구
`지금 바로 송장 입력 및 출고 진행이 가능합니다.`

### 허용 액션
- 송장 입력
- 일괄 출고 관련 작업
- 상세 보기
- 재확인

---

## 2. 송장 대기
### 의미
출고는 가능하지만 송장 입력 또는 반영이 아직 필요한 상태

### 라벨
`송장 대기`

### 설명 문구
`출고는 가능하지만 송장 입력이 아직 필요합니다.`

### 허용 액션
- 송장 입력
- 일괄 송장 반영
- 상세 보기
- 재확인

---

## 3. 보류
### 의미
자동 차단까지는 아니지만 사람이 한 번 더 확인해야 해서 임시로 멈춘 상태

### 라벨
`보류`

### 설명 문구
`발송 전 확인이 필요하여 임시로 멈춘 상태입니다.`

### 허용 액션
- 상세 보기
- 보류 해제
- 재확인
- 메모 추가

---

## 4. 차단
### 의미
지금은 보내면 안 되는 상태

### 라벨
`차단`

### 설명 문구
`취소·반품·교환·출고중지 등으로 현재 출고할 수 없습니다.`

### 허용 액션
- 상세 보기
- 사유 확인
- 재확인
- 담당자 검토

### 금지 액션
- 송장 입력
- 일괄 출고 작업

---

## 5. 재확인 필요
### 의미
상태 판단의 신뢰도가 낮아 다시 확인이 필요한 상태

### 라벨
`재확인 필요`

### 설명 문구
`상태가 불완전하거나 충돌하여 재확인이 필요합니다.`

### 허용 액션
- 상세 보기
- 재수집
- 상태 재판정
- 작업 이력 보기

---

## 4) 상위 그룹
### 실행 가능 상태
- 출고 가능
- 송장 대기

### 실행 중지 상태
- 보류
- 차단
- 재확인 필요

이 상위 그룹은 대시보드 통계나 요약 카드, 상단 집계에서 재사용할 수 있습니다.

---

## 5) 사유 체계
상태만으로는 부족하므로, 각 상태 아래 짧은 표준 사유를 둡니다.  
테이블에는 `상태 + 사유`까지만 노출하고, 설명은 Drawer에서 확장합니다.

---

## 6) 차단 사유

### 1. 취소 요청
- 라벨: `취소 요청`
- 설명: `취소 이슈로 출고가 차단되었습니다.`

### 2. 반품·교환
- 라벨: `반품·교환`
- 설명: `반품 또는 교환 이슈로 출고가 차단되었습니다.`

### 3. 출고중지
- 라벨: `출고중지`
- 설명: `출고중지 상태이므로 송장 입력 및 발송이 불가합니다.`

### 4. CS 영향
- 라벨: `CS 영향`
- 설명: `고객 요청 또는 CS 이슈로 출고를 멈춰야 합니다.`

### 5. 수동 차단
- 라벨: `수동 차단`
- 설명: `운영자가 직접 차단 처리한 주문입니다.`

---

## 7) 보류 사유

### 1. 문의 확인
- 라벨: `문의 확인`
- 설명: `고객 문의 확인 후 출고 판단이 필요합니다.`

### 2. 주문 정보 확인
- 라벨: `주문 정보 확인`
- 설명: `주소 또는 옵션 정보 재확인이 필요합니다.`

### 3. 운영 메모
- 라벨: `운영 메모`
- 설명: `내부 메모 확인 후 진행해야 합니다.`

### 4. 예외 주문
- 라벨: `예외 주문`
- 설명: `표준 출고 흐름과 다른 예외 주문입니다.`

### 5. 수동 보류
- 라벨: `수동 보류`
- 설명: `운영자가 임시 보류 처리한 주문입니다.`

---

## 8) 재확인 필요 사유

### 1. 동기화 실패
- 라벨: `동기화 실패`
- 설명: `최신 상태를 다시 가져와야 합니다.`

### 2. 송장 반영 실패
- 라벨: `송장 반영 실패`
- 설명: `송장 상태를 다시 확인해야 합니다.`

### 3. 상태 충돌
- 라벨: `상태 충돌`
- 설명: `원본 상태와 내부 상태가 일치하지 않습니다.`

### 4. 데이터 누락
- 라벨: `데이터 누락`
- 설명: `판단에 필요한 원본 데이터가 부족합니다.`

### 5. 작업 실패 영향
- 라벨: `작업 실패 영향`
- 설명: `최근 작업 실패로 상태 재확인이 필요합니다.`

---

## 9) 화면별 노출 규칙

## 대시보드
- 카드: 상위 상태 중심
- 예: `출고 차단`, `송장 대기`, `확인 필요`
- 필요 시 카드 상세에서 사유 breakdown 표시

### 예시
`출고 차단 21건`
- 취소 요청 8
- 반품·교환 5
- 출고중지 4
- CS 영향 3
- 수동 차단 1

---

## 출고 상단 탭
초기 탭은 간결하게 유지

- 전체
- 오늘 출고
- 송장 대기
- 차단됨
- 확인 필요

세부 사유는 필터 내부로 내립니다.

---

### 예시

  - `취소 요청`
- `보류`
  - `문의 확인`
- `재확인 필요`
  - `동기화 실패`

### 예시
- `차단`
- `취소 요청`

---

## 10) 상태별 허용 액션 표

| 출고 판단 | 송장 입력 | 일괄 출고 | 보류 해제 | 재확인 | 로그 보기 |
|---|---:|---:|---:|---:|---:|
| 출고 가능 | 가능 | 가능 | 불필요 | 가능 | 가능 |
| 송장 대기 | 가능 | 가능 | 불필요 | 가능 | 가능 |
| 보류 | 제한 | 제한 | 가능 | 가능 | 가능 |
| 차단 | 불가 | 불가 | 제한 | 가능 | 가능 |
| 재확인 필요 | 제한 | 불가 | 제한 | 가능 | 가능 |

---

## 11) 시각 규칙
- 출고 가능: green
- 송장 대기: amber
- 보류: muted gray/amber
- 차단: red
- 재확인 필요: blue-gray 계열

### 중요한 구분
- 차단 = “보내면 안 됨”
- 재확인 필요 = “판단 신뢰도가 낮음”

같은 경고 느낌으로 보이면 안 됩니다.

---

## 12) 문구 원칙
### 테이블
짧은 라벨만
- 취소 요청
- 문의 확인
- 동기화 실패

### Drawer
한 줄 설명 추가
- 취소 이슈로 출고가 차단되었습니다.
- 문의 확인 후 출고 판단이 필요합니다.
- 최신 상태를 다시 가져와야 합니다.

### 툴팁 / 도움말
필요 시 더 자세한 설명 제공

---

## 13) 성공 기준
- 상태와 사유만 봐도 다음 행동이 떠오른다.
- 원본 상태를 길게 읽지 않아도 된다.
- 대시보드, 출고, Drawer가 같은 운영 언어를 사용한다.
- 일괄 작업 허용 여부가 상태에 따라 자연스럽게 제어된다.
