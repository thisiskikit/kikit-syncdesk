# 06. Visual Guidelines

## 문서 목적
이 문서는 `KIKIT SyncDesk` UI 개편에서 공통으로 적용해야 할 시각 디자인 원칙을 정의합니다.  
목표는 **실무자의 눈 피로도를 낮추고, 실수 가능성을 줄이며, 운영 상태를 더 빠르게 스캔할 수 있게 만드는 것**입니다.

---

## 1) 전체 방향
### 키워드
- Quiet
- Dense but readable
- Operational
- Low-noise
- High-signal

### 참고 질감
- Linear의 차분한 정보 밀도
- Shopify Admin의 업무용 SaaS 안정감
- 과도한 대시보드형 시각화보다 **실행 중심 정보 배치**

---

## 2) 색상 사용 원칙
### 기본 배경
- Off-white 또는 light gray 계열
- 큰 면적에는 차분한 톤 사용
- 순백색 대비를 과하게 높이지 않음

### 테두리
- 굵고 진한 border 지양
- 얇은 구분선 또는 surface elevation 사용
- 모든 블록을 박스로 감싸지 않음

### 채널 색
- 쿠팡/네이버 브랜드 컬러는 작은 badge, 라벨, 아이콘 포인트 수준으로만 사용
- 화면 전체 탭, 카드 배경, 큰 면적 색상으로 쓰지 않음

---

## 3) 상태 색상 체계
상태 색은 의미를 고정합니다.

| 상태 | 톤 |
|---|---|
| 출고 가능 | Green |
| 송장 대기 | Amber |
| 보류 | Muted Gray / Amber |
| 차단 | Red |
| 재확인 필요 | Blue-Gray |
| 중립/정보 | Gray |

### 주의
- 경고 색을 너무 많이 쓰지 않기
- 모든 상태를 강한 포화색으로 표현하지 않기
- 차단과 재확인 필요가 같은 감정으로 보이지 않게 구분하기

---

## 4) 타이포그래피 원칙
### 제목
- 과장되지 않은 굵기
- 명확한 계층 구조
- 페이지 제목 > 블록 제목 > 라벨 순서

### 본문
- 짧고 직접적인 문구 사용
- 설명은 1~2줄 이내 유지
- 불필요한 장문 카피 금지

### 숫자/주문번호
- monospace 또는 스캔 친화적 스타일 권장
- 수량, 금액, 주문번호, 시간은 정렬 일관성 유지
- 수량/금액은 우측 정렬 우선

---

## 5) 공간감과 계층 구조
### 기본 원칙
- hierarchy는 border보다 spacing으로 만든다
- 관련 정보는 가까이, 다른 레벨은 멀리
- surface 간 미세한 그림자 또는 tone difference 사용 가능

### 카드/패널
- 카드가 너무 두껍고 무거워 보이지 않게
- shadow는 미세하게
- 상태 강조는 카드 전체 도색이 아니라 배지/아이콘/짧은 라벨로

---

## 6) 테이블 시각 규칙
### 행 높이
- 너무 빽빽하지 않게
- 스캔 가능한 정도의 compact density 유지

### 열 정렬
- 주문번호 / 수량 / 시각 / 금액: 스캔 우선
- 상품명: 가변 영역
- 상태/배지: 왼쪽 또는 정렬된 컬럼 안에서 일관성 유지

### 강조
- 선택 행
- 차단 행
- 재확인 필요 행  
은 subtle highlight 사용 가능

### 금지
- 긴 텍스트 다중 노출
- 색상 과다 사용
- 아이콘 과다 사용

---

## 7) Badge / Label 원칙
### Badge
- 작고 명확하게
- 정보 1개만 담기
- 장문 금지

### 채널 Badge
- `쿠팡`, `네이버`
- 브랜드 컬러 포인트만 사용

### 상태 Badge
- 출고 판단에만 강한 의미 부여
- 상태와 사유를 같은 badge로 합치지 않음

### 사유 Label
- 배지보다 한 단계 약한 시각 강도
- 상태 아래 보조 라벨로 사용

---

## 8) Empty State 원칙
### 목적
사용자가 “데이터가 없다”가 아니라 “지금 할 일이 없다”라고 이해해야 합니다.

### 예시
- `지금 바로 처리할 출고 건이 없습니다.`
- `현재 차단된 주문이 없습니다.`
- `답변이 필요한 문의가 없습니다.`
- `재시도 필요한 작업이 없습니다.`

### 금지
- `데이터 없음`
- `항목이 없습니다`
- `0건`
같은 건조한 시스템 문구만 노출

---

## 9) Alert / Warning 원칙
### 종류
- Info
- Warning
- Critical

### 표현 방식
- 상단 strip
- inline alert
- subtle callout

### 금지
- 모든 경고를 큰 빨간 박스로 표현
- 한 화면에 경고 컴포넌트 다량 중첩
- 중요도 구분 없는 동일 톤 사용

---

## 10) Drawer / Panel 원칙
### 역할
- 목록에는 얇게
- 상세는 Drawer에서 깊게

### 구조
- 상단 요약
- 섹션별 분리
- 가장 중요한 운영 판단이 먼저
- 원본 데이터는 맨 마지막 아코디언

### 금지
- Drawer 첫 화면에 JSON 배치
- 로그성 정보가 주문 판단보다 먼저 나오는 구성

---

## 11) 버튼 원칙
### Primary
현재 화면의 핵심 액션 1개만 primary

### Secondary
보조 행동

### Tertiary / Ghost
부가 액션

### 예시
출고 화면:
- Primary: 송장 입력
- Secondary: 빠른 수집
- Tertiary: 더보기

### 금지
- 한 화면에 strong primary 3개 이상
- 레거시 액션을 주 액션처럼 보이게 배치

---

## 12) 공통 품질 기준
- 스크롤 첫 화면에서 핵심 행동이 보인다.
- 색보다 구조로 우선순위를 표현한다.
- 차단/실패/확인 필요는 눈에 띄되 화면 전체를 시끄럽게 만들지 않는다.
- 채널은 보조 신호로만 존재한다.
- 제품 전체가 차분하고 실무적인 인상을 준다.
