# 01. Product Reframing

## 문서 목적
이 문서는 `KIKIT SyncDesk` UI 개편의 최상위 해석 문서입니다.  
이 문서를 통해 제품이 **무엇인지**, **무엇이 아닌지**, **이번 UI 개편이 어떤 오해를 바로잡아야 하는지**를 고정합니다.

---

## 1) 제품 한 줄 정의
**KIKIT SyncDesk는 주문 수집부터 출고, 송장, 클레임, 문의, 작업로그까지 처리하는 플랫폼 운영 데스크입니다.**

---

## 2) 이 제품은 무엇인가
이 제품은 다음 업무를 하나의 흐름으로 다루는 운영용 백오피스입니다.

- 채널 주문 수집
- 출고 가능 여부 판단
- 송장 입력 및 반영
- 클레임 / 문의 / 출고중지 대응
- 실패 작업 추적 및 복구
- 채널 원본 상태 및 연결 상태 확인

즉, 이 제품의 핵심은 **데이터를 많이 보여주는 것**이 아니라,  
**운영자가 오늘 해야 할 일을 더 안전하고 빠르게 처리하게 만드는 것**입니다.

---

## 3) 이 제품은 무엇이 아닌가
다음 해석은 이번 UI 개편에서 명시적으로 버립니다.

### 버릴 해석
- 가격 변경 툴
- 상품 편집 툴
- 채널별 원본 데이터를 늘어놓는 관리자 콘솔
- 기능이 많은 올인원 툴
- 개발자 친화적인 내부 디버그 도구

### 이유
현재 제품은 실질적으로 배송/CS/작업 실행 중심으로 진화해 왔고,  
과거 가격/상품 관련 레거시 노출이 오히려 핵심 동선을 방해하고 있습니다.

---

## 4) 현재 제품의 문제 정의
기존 구조는 아래 문제를 가지고 있습니다.

### 문제 1. 채널 중심 구조
사용자는 “쿠팡에서 무엇을 해야 하는지”보다 “지금 출고를 해도 되는지”를 먼저 알고 싶습니다.  
그러나 기존 구조는 채널 메뉴 안에서 기능을 찾아다니게 만들어, 업무 목적 중심 사고를 방해합니다.

### 문제 2. 기능 나열형 UI
다양한 기능이 존재하지만, 어떤 기능이 오늘 가장 중요한지 우선순위가 드러나지 않습니다.  
그 결과 실무자는 **지금 당장 해야 할 일**을 빠르게 파악하기 어렵습니다.

### 문제 3. 레거시 노출
`bulk-price`, `product-edit`, `grouped products` 같은 과거 기능 흔적이 메인 동선에 남아 있으면 제품 정체성이 흐려집니다.

### 문제 4. 개발자형 정보 노출
원본 상태, 내부 처리 결과, 긴 CS 이력, 로그성 정보가 메인 목록에 섞이면 운영자가 봐야 할 핵심 판단 정보가 묻힙니다.

---

## 5) 리디자인 이후의 목표 상태
UI 개편 이후 사용자는 10초 안에 아래처럼 느껴야 합니다.

### 실패한 인상
“쿠팡/네이버 기능이 섞인 관리자 콘솔이네.”

### 성공한 인상
“아, 오늘 출고·CS·실패 작업을 처리하는 운영 데스크구나.”

---

## 6) 이번 차수에서 고정할 제품 해석
아래 문장은 이후 모든 문서와 구현에서 유지되어야 합니다.

### 고정 문장
- 이 앱의 중심은 **출고 / CS / 작업 실행**이다.
- 채널은 주인공이 아니라 **데이터 출처와 세부 설정 레이어**다.
- 대시보드는 분석 화면이 아니라 **오늘의 운영 시작점**이다.
- 출고 화면은 단순 목록이 아니라 **판정 + 실행 + 예외 확인 화면**이다.
- CS는 별도 메뉴이면서 동시에 출고 판단에 침투하는 **운영 영향 레이어**다.
- 작업센터는 로그 감상 공간이 아니라 **실패 작업 복구 공간**이다.

---

## 7) 이번 차수의 범위
### 포함
- 앱 내 운영 데스크 정체성 재정의
- IA 재구성
- Dashboard 리디자인
- Fulfillment 리디자인
- 출고 판단 상태/사유 체계 적용
- 레거시 노출 제거
- 공통 시각 규칙 정리
- 관련 문서 업데이트

### 제외
- 전체 디자인 시스템 전면 재작성
- 모바일 완성도까지 포함한 전면 반응형 개편
- CS 전체 상세 화면의 풀 리디자인
- 통계/정산 화면 고도화
- 상품/가격 관리 기능 고도화

---

## 8) 핵심 사용자와 행동
### 주요 사용자
- 운영 담당자
- 출고 담당자
- CS 담당자
- 실패 작업 복구 담당자
- 채널 연결 상태를 확인하는 관리자

### 가장 자주 하는 행동
1. 오늘 처리할 출고 건 파악
2. 차단/보류/재확인 필요 건 확인
3. 송장 입력 및 반영
4. 문의/클레임으로 출고 영향 있는 건 확인
5. 실패한 수집/반영 작업 재시도
6. 채널 원본 상태와 연동 상태 확인

---

## 9) UI 개편의 핵심 질문
이번 개편은 항상 아래 질문에 답해야 합니다.

1. 이 화면은 “무엇을 볼까”가 아니라 “무엇을 할까”를 안내하는가?
2. 위험 건이 정상 건보다 더 먼저 보이는가?
3. 채널 정보는 주인공이 아니라 보조 레이어로 내려갔는가?
4. 목록은 얇고, 상세는 깊게 분리되었는가?
5. 레거시 기능 흔적이 메인 동선을 방해하지 않는가?

---

## 10) 최종 기준
이번 개편의 성공 기준은 **예쁜 UI**가 아니라 아래 상태입니다.

- 사용자가 대시보드에서 바로 행동을 시작할 수 있다.
- 출고 화면에서 “지금 보내도 되는지”를 빠르게 판단할 수 있다.
- 차단/보류/재확인 필요가 강하게 드러난다.
- 깊은 원본 정보는 필요할 때만 열린다.
- 메인 동선에서 과거 상품/가격 툴 흔적이 사라진다.
