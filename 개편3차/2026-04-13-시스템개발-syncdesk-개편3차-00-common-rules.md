# 공통 운영 규칙 — SyncDesk 개편 3차

## 목적
이 문서는 `kikit-syncdesk`의 `codex/dev` 브랜치에서 개편 3차 작업을 수행할 때 모든 작업 프롬프트 앞에 공통으로 적용할 운영 규칙입니다.

## 3차의 위치
개편 1차는 운영 데스크 IA를 재배치했고,
개편 2차는 출고/작업센터 구조를 안정화하는 차수입니다.

개편 3차는 그 다음 단계로,
**운영 데스크의 연결 완성도와 실제 운영 하드닝을 높이는 차수**입니다.

즉, 이번 차수의 핵심은:
1. 출고 ↔ CS ↔ 작업센터의 연결을 더 분명히 만들기
2. 채널/설정 허브의 진입 구조를 정리하기
3. 성능/배포/운영 검증 체계를 더 안전하게 만들기

## 공통 규칙
- 한국어 UI를 유지합니다.
- 현재 기준 브랜치는 `codex/dev`입니다.
- 이번 차수의 목표는 **운영 연결 완성 + 운영 하드닝**입니다.
- 코드 변경과 문서 변경은 반드시 같은 작업 세트로 처리합니다.
- 아래 문서는 작업 시 항상 함께 갱신합니다.
  - `docs/current-status.md`
  - `docs/change-log.md`
  - `docs/structure-overview.md`
- 검증하지 못한 내용은 완료로 단정하지 말고 반드시 `추정`으로 표시합니다.
- 기존 동작을 함부로 깨지 않습니다.
- route, API shape, data shape는 꼭 필요한 경우가 아니면 유지합니다.
- 얇은 이행 구조와 deep-link 호환을 우선합니다.
- 디자인 polish보다 먼저 operator 행동 흐름과 운영 안전성을 확인합니다.
- 구현 완료 전 반드시 아래를 남깁니다.
  - 변경 파일 목록
  - 구조 변경 요약
  - 테스트/검증 결과
  - 직접 검증 못 한 항목
  - 남은 위험
  - docs 반영 내용

## 3차 범위
이번 개편 3차의 허용 범위는 아래 3축입니다.
1. 출고-작업센터-CS 연결 흐름 강화
2. 채널/설정 허브 정리 및 레거시 진입 구조 정돈
3. 성능/배포/운영 하드닝

## 3차 금지사항
- 상품/가격 기능 대확장
- CS 전체 신규 백엔드 시스템 구축
- 모바일 완성형 대응
- 전면적인 디자인 토큰 재설계
- unrelated page 리디자인
- 대규모 DB 스키마 변경

## 작업 원칙
### 원칙 1. 허브는 예쁘게가 아니라 빠르게
채널/설정/CS 허브는 화려함보다 진입 속도와 맥락 전달이 중요합니다.

### 원칙 2. 연결은 deep-link를 깨지 않고 강화
기존 화면을 버리기보다, 운영자가 지금 맥락에서 다음 화면으로 자연스럽게 넘어가게 만드는 것이 우선입니다.

### 원칙 3. 하드닝은 기능보다 우선
운영툴은 작은 기능 추가보다 배포/성능/검증 안정화가 더 큰 가치가 있습니다.

## 응답 형식
각 작업 완료 보고는 아래 순서로 작성합니다.
1. 변경한 파일 목록
2. 무엇을 왜 바꿨는지
3. 유지된 동작 / 바뀐 내부 구조
4. 테스트 및 검증 결과
5. 직접 검증하지 못한 항목
6. 남은 리스크
7. docs 반영 내용
